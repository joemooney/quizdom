# Session Log and Promotion Path

<!-- trace:STORY-15 | ai:codex -->

quizdom keeps raw per-user exploration outside the shared AIDA knowledge graph.
The session log is the local, append-friendly record of what happened in one
user's belief exploration. AIDA receives only promoted domain objects: notable
beliefs, reusable questions, and definitions that should become shared
knowledge.

This follows the promote-only rule from ADR-12 and the graph schema in
[`graph-schema.md`](graph-schema.md). Promotion uses `BELIEF`, `TERM`, and `Q`
nodes, quizdom's custom edge vocabulary, and `weight:N` tags. There is no
separate numeric weight field.

## Storage Shape

Store one session per file under a future per-user data directory. The v1 format
is line-oriented JSONL so the session can be appended safely after every turn.
Each line is an event object with a stable `event_id`.

Recommended path shape:

```text
data/users/<user-id>/sessions/<session-id>.jsonl
```

The path is intentionally outside AIDA. If a later implementation chooses a
different private directory, the event fields below remain the compatibility
contract.

## Event Types

| Event type | Purpose |
|---|---|
| `session_started` | Opens a traversal from a seed `Q` node. |
| `question_presented` | Records the graph question shown to the user. |
| `answer_recorded` | Records the user's raw answer and normalized answer shape. |
| `term_interpreted` | Records how the user understood a loaded term in context. |
| `belief_candidate` | Captures a normalized proposition that may later be promoted. |
| `next_question_selected` | Records the deterministic or LLM-assisted traversal choice. |
| `belief_promoted` | Records the AIDA object and edges created from a candidate. |
| `session_ended` | Closes the session and records summary metadata. |

All timestamps are UTC ISO-8601 strings. IDs are opaque strings generated by the
application.

## Event Fields

Common fields:

| Field | Required | Meaning |
|---|---|---|
| `event_id` | yes | Stable event ID within this log. |
| `event_type` | yes | One of the event types above. |
| `occurred_at` | yes | UTC timestamp. |
| `session_id` | yes | Stable session ID. |
| `user_id` | yes | Local user identifier, not promoted into shared AIDA objects. |
| `turn` | no | Zero-based interaction turn for traversal events. |

Traversal fields:

| Field | Applies to | Meaning |
|---|---|---|
| `question_ref` | question and answer events | AIDA ID of the `Q` node, when known. |
| `question_text` | question events | Prompt text shown to the user. |
| `answer_mode` | question and answer events | `yes-no`, `choice[...]`, or `free-text`. |
| `raw_answer` | answer events | Exact user answer. |
| `normalized_answer` | answer events | App-normalized value used for branching. |
| `selected_next_question_ref` | selection events | AIDA ID of the next `Q`, when known. |
| `selection_reason` | selection events | Short reason for the traversal choice. |

Promotion-candidate fields:

| Field | Applies to | Meaning |
|---|---|---|
| `candidate_id` | candidate events | Stable local ID for the promotable belief. |
| `source_event_ids` | candidate events | Events used to derive the proposition. |
| `proposition` | candidate events | Normalized belief claim. |
| `topic_tags` | candidate events | Topic tags to carry into AIDA, such as `topic:free-will`. |
| `term_refs` | candidate events | Related `TERM` nodes that define loaded terms. |
| `stance_edges` | candidate events | Proposed `agrees`, `disagrees`, or `contradicts` edges. |
| `promotion_weight` | candidate events | Integer `0` through `100`; encoded as `weight:N` on promotion. |
| `promotion_reason` | candidate events | Why this raw answer is worth promoting. |

`promotion_weight` is an in-log computation aid only. The promoted AIDA object
must encode it as a `weight:N` tag.

## Promotion Rules

Promotion is a deliberate operation. It should happen after a session turn or
review pass identifies a belief worth sharing beyond the local user log.

1. Select a `belief_candidate` event.
2. Confirm the candidate has a normalized `proposition`.
3. Resolve every `term_refs` entry to an existing `TERM` node, or create a
   `TERM` node first if the definition is reusable.
4. Create a `BELIEF` object in AIDA.
5. Encode topic and selection metadata as tags, including `weight:N`.
6. Add edges from the new `BELIEF` to existing beliefs or questions using the
   schema vocabulary.
7. Record the resulting AIDA object ID back in the local log with a future
   `belief_promoted` event.

Do not promote `user_id`, raw private wording that is not needed for the shared
proposition, or the full session transcript.

## Field Mapping to AIDA

| Session field | AIDA target |
|---|---|
| `proposition` | `BELIEF` title. |
| `source_event_ids` | Description provenance, scoped to the local session ID. |
| `raw_answer` and `raw_definition` | Description evidence, summarized rather than copied wholesale when privacy matters. |
| `topic_tags[]` | Tags copied as-is. |
| `promotion_weight` | `weight:N` tag. |
| `term_refs[]` | Description references to definitions that disambiguate the belief. |
| `stance_edges[]` | AIDA custom relationship edges such as `agrees`, `disagrees`, or `contradicts`. |
| `selected_next_question_ref` | Optional `begets` edge from the `BELIEF` to the next useful `Q`. |
| `promotion_reason` | AIDA comment or description note. |

The promoted `BELIEF` description should include:

- `Source session`: local session ID and candidate ID.
- `Normalized proposition`: the exact proposition promoted.
- `Evidence`: short summary of the answer events that support it.
- `Definitions`: related `TERM` nodes that disambiguate loaded terms.

## Round-Trip Example

Input candidate from the sample log:

```json
{
  "candidate_id": "cand-001",
  "session_id": "sess-freewill-001",
  "source_event_ids": ["evt-003", "evt-004"],
  "proposition": "Free will requires genuine alternative possibilities",
  "topic_tags": ["topic:free-will"],
  "term_refs": ["TERM-free-will-libertarian"],
  "stance_edges": [
    {
      "edge": "disagrees",
      "target_ref": "BELIEF-free-will-compatible-determinism"
    }
  ],
  "promotion_weight": 55
}
```

Promoted AIDA object:

```yaml
id: BELIEF-free-will-alternative-possibilities
type: BELIEF
title: Free will requires genuine alternative possibilities
tags:
  - topic:free-will
  - weight:55
description: |
  Source session: sess-freewill-001 / cand-001.
  Normalized proposition: Free will requires genuine alternative possibilities.
  Evidence: The user answered yes to the free-will seed question and clarified
  free will as the ability to choose differently in the exact same situation.
  Definitions: TERM-free-will-libertarian.
relationships:
  - type: disagrees
    target: BELIEF-free-will-compatible-determinism
```

Round-trip record appended to the local log:

```jsonl
{"event_id":"evt-008","event_type":"belief_promoted","occurred_at":"2026-05-31T15:08:00Z","session_id":"sess-freewill-001","user_id":"local-joe","candidate_id":"cand-001","promoted_ref":"BELIEF-free-will-alternative-possibilities","promoted_tags":["topic:free-will","weight:55"],"promoted_edges":[{"edge":"disagrees","target_ref":"BELIEF-free-will-compatible-determinism"}]}
```

After round-trip, the local log remains the audit trail for how the belief was
derived, while AIDA holds only the reusable graph object and its typed edges.
